#  Установка Elasticsearch

В этом разделе описана установка и базовая настройка Elasticsearch на виртуальной машине `elk-lab`.

### Примечания перед началом

- Elasticsearch 9.x поставляется со встроенным OpenJDK. Устанавливать Java отдельно не нужно.
- Настраиваем Elasticsearch как **одноузловой кластер** в учебных целях.
- По умолчанию после установки Elasticsearch слушает только `localhost`. Настроим его так, чтобы к нему можно было подключаться с других машин в сети.

[Официальная документация](https://www.elastic.co/docs/deploy-manage/deploy/self-managed/install-elasticsearch-with-debian-package#deb-repo)


## 1. Импорт ключа и добавление репозитория:
Сначала обновите систему и установите утилиты для работы с HTTPS-репозиториями. Затем добавьте официальный GPG-ключ Elastic и сам репозиторий в список источников `apt`.
```bash
# Обновление пакетов и установка зависимостей
sudo apt update && sudo apt upgrade -y
sudo apt install -y apt-transport-https curl wget gnupg

# Импорт PGP-ключа Elastic
wget -qO - https://artifacts.elastic.co/GPG-KEY-elasticsearch | sudo gpg --dearmor -o /usr/share/keyrings/elasticsearch-keyring.gpg

# Добавление репозитория Elasticsearch 9.x (block Russia)
# Заблокирован в России, используется VPN или зеркало Яндекс ниже
echo "deb [signed-by=/usr/share/keyrings/elasticsearch-keyring.gpg] https://artifacts.elastic.co/packages/9.x/apt stable main" | sudo tee /etc/apt/sources.list.d/elastic-9.x.list

# Яндекс зеркала
echo "deb [trusted=yes] https://mirror.yandex.ru/mirrors/elastic/9/ stable main" | sudo tee /etc/apt/sources.list.d/elastic-9.x.list
```

**Важно**: Во время установки в терминале отобразится **автоматически сгенерированный пароль для суперпользователя elastic**. Обязательно скопируйте и сохраните его в надежном месте - он понадобится для подключения к кластеру.

## 2. Установка Elasticsearch:
После обновления индекса пакетов установите Elasticsearch .
```bash
sudo apt-get update && sudo apt-get install elasticsearch
```

## 3. Настройка работы в сети:
По умолчанию Elasticsearch слушает только `localhost`. Чтобы к нему могли подключаться другие узлы в учебной сети, нужно изменить конфигурацию. Откройте файл `/etc/elasticsearch/elasticsearch.yml` в редакторе (например, `sudo nano /etc/elasticsearch/elasticsearch.yml`) и укажите:
```yaml
# ================================================
# НАСТРОЙКИ ДЛЯ УЧЕБНОЙ СЕТИ (ОБЯЗАТЕЛЬНЫЕ)
# ================================================

# Адрес, на котором Elasticsearch будет принимать HTTP соединения
# 0.0.0.0 - слушать все интерфейсы (для доступа из сети)
network.host: 0.0.0.0

# HTTP порт для REST API
http.port: 9200

# ================================================
# НАСТРОЙКИ ДЛЯ УЧЕБНЫХ ЦЕЛЕЙ (НЕОБЯЗАТЕЛЬНЫЕ)
# ================================================

# Имя кластера (должно быть одинаковым для всех узлов)
cluster.name: my-study-cluster

# Уникальное имя узла
node.name: node-1

# Режим одноузлового кластера (ОБЯЗАТЕЛЕН для одного сервера)
discovery.type: single-node
```

**Примечание**: Для кластера из нескольких узлов настройка сложнее и требует указания `discovery.seed_hosts` и `cluster.initial_master_nodes`.

**Возможная ошибка**: Если на следующем шаге *Запуск и добавление в автозагрузку* возникает ошибка, то скорее всего вам надо закоментировать строку: `#cluster.initial_master_nodes: ["mmm-VirtualBox"]`. Она не совместима с `discovery.type: single-node`.

## 4. Настройка heap size
Ограничьте использование оперативной памяти Elasticsearch:
```bash 
sudo nano /etc/elasticsearch/jvm.options
```
Найдите и измените строки:
```text
-Xms4g
-Xmx4g
```
Это ограничит прожорливость Elasticsearch до 4Gb

## 5. Запуск и добавление в автозагрузку:
Включите и запустите службу Elasticsearch, а затем проверьте её статус.
```bash
sudo systemctl daemon-reload
sudo systemctl enable --now elasticsearch.service
sudo systemctl status elasticsearch.service
```

## 6. Что Elasticsearch настроил автоматически
Когда вы установили Elasticsearch из Debian-пакета(`apt`), он сам:
- Сгенерировал сертификаты `TLS/SSL`
    - Сертификат для `HTTP` слоя: `/etc/elasticsearch/certs/http_ca.crt`
    - Сертификаты для транспортного слоя: `/etc/elasticsearch/certs/`

- Настроил `HTTPS` на порту `9200`. Теперь Elasticsearch слушает только защищённые соединения
- Создал суперпользователя `elastic` и сгенерировал для него пароль
- Настроил все пути к сертификатам в конфигурации

## 7. Запуск
```bash
sudo curl --cacert /etc/elasticsearch/certs/http_ca.crt -u elastic:PASSWORD_HERE https://localhost:9200
```
- `--cacert /etc/elasticsearch/certs/http_ca.crt` указывает `curl` доверять сертификату Elasticsearch
- `-u elastic:PASSWORD_HERE` логин и пароль для аутентификации
- `https://localhost:9200` адрес с HTTPS (не HTTP)

Если все успешно, видим ответ:

Если будете проверять с другой машины в сети, используйте:
```bash 
# Узнайте IP вашего сервера, к примеру так
ip a | grep "inet " | grep -v 127.0.0.1

# С другого компьютера
curl --insecure -u elastic:PASSWORD_HERE https://192.168.1.100:9200
```
**Важно:**
- `--insecure` используется только потому, что на другой машине нет вашего сертификата
- В production нужно скопировать `/etc/elasticsearch/certs/http_ca.crt` на клиентскую машину
